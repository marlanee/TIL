#### Git Workflow
what is Git workflow?

- GIt workflow: Github를 이용해 협업하는 방법
- Feature Branch Workflow: Shared repository model.
  - 각 사용자가 Github의 소유권을 공유 받는 방식
- 중앙 원본 저장소 개발 X
- 기능 개발은 별도의 branch에서 진행
- master branch push X
  - git push origin "나의 branch"
  - "나의 branch"에 개발 후 master에 병합
- 작업 완료시 branch push, Pull request
- Pull request: branch 작업 내용을 master branch에 merge 요청하는 기능
  - PR의 목적: Code Review
  - 관리자가 코드 검토 후 merge 진행
---
#### Branch Workflow 과정
  - 구성원이 git clone으로 복제
  - 작업 후 branch 업로드
    - Pull request 생성됨
  - Code Review 이후 Master에 병합
  - Master branch로 이동 후 merge된 내용을 git pull
  - merge된 local branch 삭제
  - brnach 생성 후 반복
  ---
  - 신뢰할 수 있는 팀원 몇몇과 진행할 때 사용하는 방법

### Fork
---
- 오픈소스가 개발되어왔던 방식
- Fork로 PR 승인시 Contributor에 등록됨
---
- 중앙 원본 저장소에 직접 push 불가시 사용
- 내 repository에 fork로 원본을 복제
- 복제한 원본을 local에 clone으로 복제
- 원본의 주소도 local에 추가함
  - pull을 하기 위함임
- 작업 후 복제 원본에 push
- 복제 원본 master에서 merge
- 복제 원본에서 원본으로 PR 진행

### Unit Test
---
- 컴퓨터에게 기능 단위로 검증시키는 방법
- 검증 코드를 만들어야함
- 프롬프트 입력 꿀팁: AAA패턴으로 기능 테스트 추가
---
- AI는 코드를 완벽하게 작성할까?
  - 오류 발생: 여러 번의 기능 추가 및 수정시
    - 기존 기능 파괴: 회귀 버그
  - 따라서 검증을 해야함: UNIT TEST
- 그렇다면 어떻게 검증할까?
  - 검증을 컴퓨터에게 맡기자!
    - 컴퓨터 테스트가 가능한 로직 구현 필요
- Unit test 과정: AAA 패턴
  1. 준비 단계
  2. 실행 단계
  3. 검증 단계
- 주의 사항
  - 테스트 코드를 통과해도 버그 발생 가능
  - 테스트 코드 작성에 많은 시간을 쏟지 말 것
  - 어떤 기능에 테스트 코드를 작성할지 판단 필요
- 테스트 코드를 붙여야 하는 곳
  - 포인트 계산, 게시글 삭제 조건, 로그인 및 권한 등
- Test coverage: test case가 전체 코드의 몇%를 검증하는지.
  - 높다고 좋은 것은 아님
